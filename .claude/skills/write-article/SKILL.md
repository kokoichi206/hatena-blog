# ブログ記事を書く

開発中に得た知見を、このリポジトリの規約（ファイル配置・書式契約・投稿フロー）に沿った記事にする。

## 置き場所

- 記事本文: `articles/<slug>.md`。slug は英語 snake_case（例: `edge_level_trigger.md`）。
- 図・スクリーンショット: `articles/img/` に置き、本文からは `./img/<name>.png` で参照する。
  投稿時に `scripts/upload.sh` が GitHub の raw URL へ書き換えるので、この相対パス形式を崩さない。

## 書式契約（`scripts/upload.sh` の前提）

- 1 行目: `# タイトル`、2 行目: 空行、3 行目以降: 本文。
  この形式を外すと upload がタイトル空でエラーになる。
- 導入段落の直後に `<!-- more -->` を置く（はてなブログの「続きを読む」境界）。

## トーン・構成の抽出チェックリスト

書き始める前に、テーマの近い既存記事を 2〜3 本読み、以下を合わせる。

- 文体: です・ます調。誇張せず、一人称の実体験ベースで書く。
- 構成: 目次（見出しアンカーのリスト）→「環境」（バージョン等の前提）→
  「この記事で伝えたいこと」→ 実例・本文 →「おわりに」。短い記事では目次・環境を省いてよい。
- コード例: 記事内には要点の最小片だけを貼り、全体は公開リポジトリへの
  commit hash 付き permalink でリンクする。
- 図解: 概念の対比や時系列など文章で伝わりにくい箇所は `articles/img/` に図を置いて挟む。

## 最小再現コードの置き場

- 記事のための再現コード・サンプルプロジェクトはこのリポジトリに入れず、
  別の公開リポジトリに置いて記事からリンクする。

## 投稿フロー

1. **lint を通す**

   ```bash
   npx textlint articles/<slug>.md
   ```

   - textlint の CI は PR にしか走らない。main 直コミット運用では push 前のローカル実行が唯一の検査になる。

2. **ブランチを切らず main へ直接コミットする**
   - 記事の追加・更新はブランチ・PR を作らず main に直接 commit + push する。
   - **1 push で変更する記事 md は 1 本だけにする。**
     `create-release.yml` が push の diff から変更 md を拾って `upload.sh`（引数 1 ファイル）に渡すため、
     複数 md を同時に push すると upload が壊れる。

3. **push 後の挙動を確認する**
   - main への push（`articles/**` 変更）で GitHub Actions がはてなブログへ **下書き（draft）** として投稿する。
     公開はユーザーがはてなブログ側で行う。
   - Actions の成否を確認し、記事ファイルのパスと commit リンクを報告して終える。
